## 1. 概述

`impetus.nvim` 是一个专为 Impetus / LS-DYNA 关键字输入文件打造的 Neovim 插件。它把你的编辑器变成一个功能完备的 IDE，提供以下核心能力：

- **语法高亮**：自定义暗色高对比度配色方案（品红/青色/绿色/红色强调）
- **智能补全**：关键字、参数、对象 ID、枚举选项的上下文感知补全
- **实时检查（Lint）**：三级严重程度 —— 错误（Error）、警告（Warning）、可疑（Suspicion）
- **参数代数**：定义 `%参数` 并 inline 替换或求值
- **跨文件感知**：递归解析 `*INCLUDE` 文件中的参数定义和对象引用
- **智能导航**：跳转到定义、查找引用、浏览对象注册表
- **代码折叠**：折叠关键字块和控制指令块（`~if`、`~repeat`、`~convert`）
- **侧边帮助面板**：实时显示 `commands.help` 中的参数文档
- **操作日志**：每次结构性修改都记录到 `impetus_nvim.log`

## 2. 安装与配置

### 2.1 使用 lazy.nvim 安装

```lua
{
  dir = "E:/AI/codex/case1/impetus.nvim",
  config = function()
    require("impetus").setup({
      help_file = "E:/AI/codex/case1/commands.help",
      auto_load = true,
      lint_on_save = true,
    })
  end,
}
```

### 2.2 插件加载优先级

插件按以下优先级加载关键字数据库：

1. `help_file`（如果提供且可读）
2. 缓存数据库（`cache_file` 或自动检测）
3. 内置的 `data/keywords.json`

### 2.3 文件类型自动检测

插件自动为以下文件设置 `filetype=impetus`：

- `*.k`
- `*.key`
- `commands.help`

手动设置 `filetype=impetus` 的缓冲区也会获得所有插件功能。

### 2.4 关键配置项

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `help_file` | string | `nil` | `commands.help` 文件路径 |
| `auto_load` | boolean | `true` | 启动时自动加载数据库 |
| `lint_on_save` | boolean | `true` | 保存时自动运行 Lint |
| `dev_hot_reload` | boolean | `false` | 焦点返回时热重载插件（开发模式） |
| `tab_field_jump` | boolean | `true` | 插入模式下 `<Tab>` 在参数字段间跳转 |
| `blink_menu_keys` | boolean | `true` | Blink.cmp 菜单中使用 `j/k/Space` 导航 |
| `ref_marks_enabled` | boolean | `true` | 默认启用引用标记（绿/蓝下划线） |

## 4. 核心编辑快捷键详解

### 4.1 `,c` — 注释/取消注释当前关键字块

这是你最常用的快捷键之一。它的行为是**智能切换**：根据当前状态决定是注释还是取消注释。

**工作原理：**
- 如果光标所在的关键字块**没有被注释**，按 `,c` 会把整个块注释掉（包括关键字行和所有数据行）
- 如果光标所在的关键字块**已经被注释**，按 `,c` 会恢复整个块

**举例说明：**

假设你的文件中有：

```
*NODE
1, 0.0, 0.0, 0.0
2, 1.0, 0.0, 0.0
3, 1.0, 1.0, 0.0
```

光标在 `*NODE` 行或任意数据行上，按 `,c` 后变成：

```
# *NODE
# 1, 0.0, 0.0, 0.0
# 2, 1.0, 0.0, 0.0
# 3, 1.0, 1.0, 0.0
```

再按一次 `,c`，恢复为原始状态。

**智能恢复机制（重要）：**

当块内有分隔注释（如 `# Output`、`# Parts`）时，插件会**保留**这些注释，只恢复真正的数据行。例如：

```
*FUNCTION
"Smoothed displacement"
23
smooth_d(%disp, 0, %tend)
#
# Output
```

按 `,c` 注释后：

```
# *FUNCTION
# "Smoothed displacement"
# 23
# smooth_d(%disp, 0, %tend)
# #
# # Output
```

再按 `,c` 恢复时，会正确恢复 `*FUNCTION` 的数据行，而 `# Output` 保持注释状态（因为它不是原始数据的一部分）。

**注意：** 此快捷键不处理控制指令块（`~if` / `~repeat` 等），只处理 `*关键字` 块。

---

### 4.2 `dk` — 删除（剪切）当前块

删除光标所在的关键字块或控制块，内容进入 Vim 的未命名寄存器（`"`），可以用 `p` 粘贴。

**举例：**

```
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3

*NODE
1, 0.0, 0.0, 0.0
```

光标在 `*MAT_ELASTIC` 行，按 `dk`，`MAT_ELASTIC` 块被删除，文件变成：

```
*NODE
1, 0.0, 0.0, 0.0
```

然后按 `p` 可以在光标后粘贴刚才删除的块。

**对控制块同样有效：** 光标在 `~if` 行，按 `dk` 会删除整个 `~if` / `~end_if` 结构。

---

### 4.3 `,y` — 复制当前块

复制光标所在的关键字块或控制块到寄存器，**不删除**原内容。

**举例：** 光标在 `*NODE` 块内，按 `,y`，然后用 `p` 粘贴，会在光标下方插入一个完整的 `*NODE` 块副本。

---

### 4.4 `,j` / `,k` — 移动当前块

`,j` 把当前关键字块**向下移动**一个块的位置。  
`,k` 把当前关键字块**向上移动**一个块的位置。

**举例：**

```
*NODE
1, 0.0, 0.0, 0.0

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

光标在 `*NODE` 块内，按 `,j`，文件变成：

```
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3

*NODE
1, 0.0, 0.0, 0.0
```

**注意：** 这两个快捷键只移动**关键字块**，不移动控制指令块。

---

### 4.5 `,I` — 插入关键字模板

在光标位置插入一个关键字模板。通常用于：
- 你想手动输入一个新关键字，但不想从空行开始
- 配合补全系统使用

**举例：** 按 `,I` 后输入 `MAT_ELASTIC`，会插入：

```
*MAT_ELASTIC
"optional_title"
1, 0.0, 0.0, 0.0
```

---

### 4.6 `<Tab>` — 参数字段跳转（插入模式）

当 `tab_field_jump = true`（默认）时，在插入模式下按 `<Tab>` 可以在关键字的各个数据字段之间跳转。

**举例：** 你正在编辑：

```
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

光标在 `2700.0` 位置，按 `<Tab>`，光标跳到 `70.0e9`。再按 `<Tab>`，跳到 `0.3`。

**注意：** 这要求 Blink.cmp 的 snippet 补全已经展开，且模板中包含 `<Tab>` 占位符。

## 5. 导航快捷键详解

### 5.1 `gd` — 跳转到定义

这是 IDE 级别的核心功能。光标放在以下任何内容上按 `gd`，都会跳转到其定义位置：

| 光标下的内容 | 跳转目标 |
|-------------|---------|
| `%param_name` | `*PARAMETER` 或 `*PARAMETER_DEFAULT` 中的定义行 |
| `pid`、`mid`、`gid` 等对象 ID | 该 ID 的首次定义位置（如 `*PART`、`*MAT_`、`*GEOMETRY`） |
| `fcn(id)` | `*FUNCTION` 中 `id` 的定义 |
| `crv(id)` | `*CURVE` 中 `id` 的定义 |

**举例 1 — 参数跳转：**

```
*PARAMETER
%density, 2700.0

*MAT_ELASTIC
1, %density, 70.0e9, 0.3
```

光标放在 `%density` 上，按 `gd`，光标跳转到 `*PARAMETER` 块的 `%density, 2700.0` 行。

**举例 2 — 对象 ID 跳转：**

```
*PART
"Front bumper"
1, 1, 1, 0, 0

*SECTION_SHELL
1, 1, 0.5
```

光标放在 `*SECTION_SHELL` 行第二个字段的 `1` 上（这是 `pid`），按 `gd`，跳转到 `*PART` 块的定义处。

**跨文件跳转：** 支持任意层级的 `*INCLUDE` 嵌套（如 `main.k → b.k → d.k`）。`gd` 会自动打开包含定义的文件并跳转到对应位置。

---

### 5.2 `gr` — 列出引用

光标放在定义上，按 `gr` 会列出所有引用该对象/参数的位置。

**举例：**

```
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3

*PART
"Part A"
1, 1, 1, 0, 0

*PART
"Part B"
2, 1, 2, 0, 0
```

光标放在 `*MAT_ELASTIC` 行的 `1`（`mid=1`）上，按 `gr`，弹出窗口显示：

```
References for MAT 1:
  *PART "Part A"  (pid=1)
  *PART "Part B"  (pid=2)
```

按 `j`/`k` 选择，按 `<Space>` 或 `<CR>` 跳转到对应位置。

**对 `fcn(id)` / `crv(id)` 同样有效。**

---

### 5.3 `,n` / `,N` — 关键字间跳转

`,n` 跳到**下一个**关键字行。  
`,N` 跳到**上一个**关键字行。

**举例：** 在一大段数据行中快速跳到下一个 `*MAT_` 或 `*PART` 关键字。

---

### 5.4 `%` — 匹配跳转

在控制指令结构（`~if` / `~end_if` 等）之间跳转。如果不在控制指令上，回退到 Vim 默认的括号匹配行为。

**举例：**

```
~if %use_gravity == 1
*LOAD_BODY_Z
1, 9.81
~end_if
```

光标在 `~if` 行，按 `%`，跳转到 `~end_if`。再按 `%`，跳回 `~if`。

---

### 5.5 `,m` — 跳转到匹配的控制块

同 `%`，但专门用于控制块匹配，不会回退到括号匹配。

---

### 5.6 `,b` — 检查未匹配的控制块

扫描整个文件，检查所有 `~if`、`~repeat`、`~convert`、`~scope` 是否成对出现。如果有未匹配的，会在 quickfix 中列出。

**举例：** 你写了：

```
~if %use_gravity == 1
*LOAD_BODY_Z
1, 9.81
```

忘记了 `~end_if`。按 `,b`，quickfix 显示：

```
Unmatched ~if at line 42
```

---

### 5.7 `,o` — 打开引用文件

光标放在 `*INCLUDE` 或 `*SCRIPT_PYTHON` 的文件路径上，按 `,o`，会在**左侧分屏**中打开那个文件。

**举例：**

```
*INCLUDE
../mesh/part1.k
```

光标放在 `../mesh/part1.k` 上，按 `,o`，左侧出现新窗口打开 `part1.k`。

---

### 5.8 `,O` — 在 Impetus GUI 中打开

在系统默认的 Impetus GUI 中打开当前文件。用于快速预览模型。

## 6. 折叠快捷键详解

插件使用 **manual** 折叠方法，但提供了智能的折叠状态管理。

### 6.1 `,f` — 折叠/展开所有关键字块

按一次，所有 `*关键字` 块折叠，只显示关键字行：

```
+-- 5 lines: *NODE····························································
+-- 3 lines: *MAT_ELASTIC····················································
+-- 8 lines: *PART···························································
```

再按一次，全部展开。

### 6.2 `,t` — 折叠/展开当前关键字块

只影响光标所在的关键字块，不影响其他块。

**举例：** 光标在 `*NODE` 块内，按 `,t`，只有 `*NODE` 块折叠/展开。

### 6.3 `,F` — 折叠/展开所有控制块

折叠所有控制指令结构（`~if`、`~repeat`、`~convert`、`~scope`）。

**举例：**

```
+-- 12 lines: ~if %use_gravity == 1··········································
*NODE
1, 0.0, 0.0, 0.0
+-- 8 lines: ~repeat 5·······················································
```

### 6.4 `,T` — 折叠/展开当前控制块

只影响光标所在的控制指令结构。

### 6.5 `,z` — 折叠/展开所有折叠

**一键切换所有折叠状态** —— 关键字折叠 + 控制块折叠同时切换。

**状态记忆：** 插件会记住每个折叠的打开/关闭状态，切换文件后状态会保留（在当前 Neovim 会话内）。

## 7. 帮助与信息快捷键详解

### 7.1 `,h` — 切换右侧帮助面板

打开或关闭右侧垂直分屏的帮助面板。面板显示当前关键字的数据结构签名和字段说明。

**举例：** 光标在 `*MAT_ELASTIC` 行，按 `,h`，右侧面板显示：

```
*MAT_ELASTIC
  mid    — Unique material identification number
  rho    — Density
  E      — Young's modulus
  nu     — Poisson's ratio
```

光标移动到其他关键字，面板内容自动更新。

### 7.2 `,H` — 打开完整手册

在浏览器中打开当前关键字的完整 Impetus 在线文档页面。

### 7.3 `,,` / `,R` — 引用/选项补全弹出窗口

光标放在某个数据字段上，按 `,,`，弹出窗口显示该字段所有合法的选项值。

**举例：** 光标在 `*UNIT_SYSTEM` 的 `units` 字段：

```
*UNIT_SYSTEM
SI, K
```

光标放在 `SI` 上，按 `,,`，弹出：

```
SI       — [m, kg, s]
MMTONS   — [mm, ton, s]
CMGUS    — [cm, g, μs]
IPS      — [in, slinch, s]
...（共14个选项）
```

按 `j`/`k` 选择，`<Space>` 替换当前值。

**对对象 ID 引用同样有效：** 光标在 `*PART` 的 `mid` 字段，按 `,,` 会弹出所有已定义的 `MAT` ID 列表。

### 7.4 `,i` — 切换信息面板

打开信息面板，显示模型统计树：

```
Model Statistics
├── Nodes:        15420
```

## 9. Clean 清理命令详解

Clean 命令用于整理和规范化 Impetus 输入文件。共有 4 种模式：

| 命令 | 删除注释 | 删除空行 | 删除未知块 | 对齐参数 | 美化排版 | 说明 |
|------|----------|----------|------------|----------|----------|------|
| `:clean` | ❌ | ❌ | ❌ | ❌ | ❌ | 只清除标记和诊断 |
| `:clean -c` | ✅ | ✅ | ❌ | ❌ | ❌ | 温和清理 |
| `:clean -a` | ✅ | ✅ | ✅ | ✅ | ✅ | 深度清理 + 美化 |
| `:clean -s` | ❌ | ❌ | ❌ | ❌ | ✅ | 只美化对齐（不删除内容） |

### 9.1 `:clean`（无参数）— 清除标记和诊断

**功能：** 不修改文件内容，只清除两类临时标记：
1. 控制指令对的视觉标记（如 `~if`/`~end_if` 之间的彩色高亮背景）
2. Lint 诊断信息（虚拟文本、标记列）

**适用场景：** 你想清除屏幕上的彩色标记和错误提示，但不想改文件内容。

**举例：**

运行 `:Ccheck` 后，文件中出现了很多红色/黄色的错误提示。按 `:clean`，所有提示消失，文件内容不变。

---

### 9.2 `:clean -c`（Warm Clean）— 温和清理

**功能：** 删除文件中的"噪音"内容，但**保留所有关键字块**（包括未知关键字）。具体删除：

1. **注释行**（`#` 或 `$` 开头的行）
2. **空行**（块内的前导和尾随空行）
3. **仅包含逗号的占位行**（如 `, , ,`）—— 只在单行数据块中删除
4. **文件开头在第一个关键字之前的注释/空行**

**同时会：** 把关键字行的小写转为大写（如 `*node` → `*NODE`）。

**举例：**

**清理前：**

```
# This is the main input file
# Created by Eric

*NODE
# Define nodes
1, 0.0, 0.0, 0.0

2, 1.0, 0.0, 0.0


*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3

*MY_CUSTOM_COMMAND
1, 2, 3
```

**执行 `:clean -c` 后：**

```
*NODE
1, 0.0, 0.0, 0.0
2, 1.0, 0.0, 0.0
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
*MY_CUSTOM_COMMAND
1, 2, 3
```

**注意：**
- `# This is the main input file` 等头部注释被删除
- `# Define nodes` 等块内注释被删除
- `*NODE` 块内的空行被删除
- `*MY_CUSTOM_COMMAND` 是未知关键字，但**被保留**（这是 `-c` 和 `-a` 的核心区别）

**适用场景：** 你想清理注释和空行，但**不想丢失任何自定义关键字或数据**。

---

### 9.3 `:clean -a`（Full Clean）— 深度清理

**功能：** `-a` 会**顺序执行四个操作**，每一步都在前一步的结果上继续：

**Step 1: Warm Clean（同 `-c`）**
> 删掉所有注释、空行、占位行。

**Step 2: Advanced Clean（激进清理 — `-a` 独有）**
> 遍历所有关键字块，**删除不在 `keywords.json` 中定义的未知关键字块整段**。
>
> 例如你文件里有自定义关键字 `*MY_CUSTOM_COMMAND`，`-a` 会把它及其所有数据行整段删除。

**Step 3: Align Parameter Blocks（参数对齐）**
> 找到所有 `*PARAMETER` / `*PARAMETER_DEFAULT` / `*CURVE` / `*TABLE` / `*PATH` / `*OBJECT` 块，把参数定义和数据行按列对齐。

**Step 4: Simple Beautify（简单美化 — 同 `-s`）**
> 在不删除任何内容的前提下，规范化所有数据行的空格、运算符间距、关键字大小写，以及 `~repeat` 控制块的缩进。

**举例：**

**清理前：**

```
# Main file

*NODE
1, 0.0, 0.0, 0.0
2, 1.0, 0.0, 0.0

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3

*MY_CUSTOM_COMMAND
1, 2, 3

*PARAMETER
%density, 2700.0
%young, 70.0e9
%poisson, 0.3
```

**执行 `:clean -a` 后：**

```
*NODE
1, 0.0, 0.0, 0.0
2, 1.0, 0.0, 0.0
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
*PARAMETER
%density,  2700.0
%young,    70.0e9
%poisson,  0.3
```

**发生了什么：**
1. 所有注释和空行被删除
2. `*MY_CUSTOM_COMMAND` 整段被删除（未知关键字）
3. `*PARAMETER` 块被对齐，`%` 后的逗号对齐到同一列
4. 所有数据行进行空格规范化、关键字大小写统一、`~repeat` 缩进调整

**适用场景：**
- 你确认文件里没有需要保留的自定义关键字
- 你想得到一份"标准、干净、对齐"的最终版本提交给求解器
- **注意：此操作不可逆！** 建议先保存副本或确认版本控制已提交

**通知信息：**

执行后会显示类似：

```
Impetus clean -a done. Removed: 15 (warm=8 adv=7), aligned: 3, beautified: 12
```

表示共删除了 15 行（其中 8 行来自温和清理，7 行来自未知块删除），对齐了 3 行参数定义，美化了 12 行数据。

---

### 9.4 `:clean -s`（Simple Beautify）— 简单美化

**功能：** **不删除任何内容**，只调整排版：
- 数据行的列对齐
- 空格规范化
- 关键字行的大小写规范化
- `*PARAMETER` / `*PARAMETER_DEFAULT` 参数定义的特殊美化：等号对齐、双引号对齐、逗号后固定 3 个空格、引号后的尾随字段之间也补空格

**举例：**

**美化前：**

```
*NODE
1,   0.0,    0.0,   0.0
2, 1.0, 0.0, 0.0
3,   1.0, 1.0,   0.0
```

**执行 `:clean -s` 后：**

```
*NODE
1, 0.0, 0.0, 0.0
2, 1.0, 0.0, 0.0
3, 1.0, 1.0, 0.0
```

**负号（一元减号）的特殊处理：**

`clean -s` 会规范化运算符周围的空格，但对**一元负号**（如 `-0.5`）会做特殊保护：

| 输入 | 输出 | 说明 |
|------|------|------|
| `D, X, 23, -0.5` | `D, X, 23, -0.5` | ✅ 保留逗号后的空格 |
| `a - b` | `a-b` | ✅ 二元减法去空格 |
| `%x - %y` | `%x-%y` | ✅ 参数表达式去空格 |
| `( -1 )` | `( -1 )` | ✅ 一元负号保留空格 |

**适用场景：** 文件内容都是对的，只是排版不整齐，想快速美化。

## 10. Replace 替换命令详解

Replace 命令用于将参数引用（`%param`）替换为实际值，并可选择性地对表达式求值。

| 命令 | 替换参数引用 | 求值表达式 | 替换定义行 | 内建函数 | 说明 |
|------|-------------|-----------|-----------|----------|------|
| `:re` | ✅ | ❌ | ❌ | ❌ | 纯文本替换 |
| `:re -a` | ✅ | ✅ | ❌ | ❌ | 替换 + 算术求值 |
| `:re -b` | ✅ | ✅ | ✅ | ✅ | 全量替换 + 内建函数 |
| `:re -c` | ✅ | ✅ | ✅ | ✅ | 同 `-b`，带循环溢出检测 |

### 10.1 `:re` — 纯文本替换

**功能：** 把所有 `%param` 替换为 `*PARAMETER` 中定义的值，**不做任何数学运算**。

**举例：**

**替换前：**

```
*PARAMETER
%density, 2700.0
%young, 70.0e9

*MAT_ELASTIC
1, %density, %young, 0.3
```

**执行 `:re` 后：**

```
*PARAMETER
%density, 2700.0
%young, 70.0e9

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

**注意：**
- `*PARAMETER` 定义行**保持不变**（`%density` 还在）
- 只有引用 `%density` 的地方被替换
- 如果参数值本身是表达式（如 `%size, 10 + 5`），替换后仍然是 `10 + 5`，不会计算成 `15`

**适用场景：** 你只想把参数名称替换成字面量，保留表达式的原始形式。

---

### 10.2 `:re -a` — 替换 + 算术求值

**功能：** 在 `:re` 的基础上，对替换后的表达式进行算术求值。支持的运算符：`+` `-` `*` `/` `^` `(` `)`。

**举例：**

**替换前：**

```
*PARAMETER
%size, 10
%offset, 5

*NODE
1, %size + %offset, 0.0, 0.0
```

**执行 `:re -a` 后：**

```
*PARAMETER
%size, 10
%offset, 5

*NODE
1, 15, 0.0, 0.0
```

**发生了什么：**
1. `%size` 替换为 `10`，`%offset` 替换为 `5`
2. `10 + 5` 被求值为 `15`

**注意：**
- `*PARAMETER` 定义行仍然保持不变
- 只支持基本算术，**不支持** `sin`、`sqrt` 等数学函数

**`*FUNCTION` 块的特殊处理：**

`*FUNCTION` 表达式行会执行**部分求值**：`%param` 引用会被替换，纯数值子表达式（如 `0.5*0.01`）会被求值为单个数字，但求解器变量（`t`、`x`、`y`、`z`）和自定义函数调用（`smooth_d`、`crv`、`fcn`）会原样保留。参数值如果使用科学计数法，也会在符号表达式中继续保持科学计数法，例如 `500e6` 不会被改写成 `500000000`。

**举例：**

```
替换前：smooth_d(12000, 0, 0.5*%tend)
替换后：smooth_d(12000, 0, 5e-05)
```

**适用场景：** 你的参数表达式只涉及加减乘除，想直接得到数值结果。

---

### 10.3 `:re -b` — 全量替换 + 内建函数求值（最强大）

**功能：** `:re -b` 会做三件事：
1. 替换所有参数引用（同 `:re`）
2. 求值算术表达式（同 `:re -a`）
3. **同时替换 `*PARAMETER` 定义行本身**（把定义也变成数值）
4. **支持内建数学函数**（`sin`、`cos`、`sqrt` 等 30+ 个函数）

**举例：**

**替换前：**

```
*PARAMETER
%pi, 3.14159265
%radius, 10.0
%area, %pi * %radius^2

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

**执行 `:re -b` 后：**

```
*PARAMETER
%pi, 3.14159265
%radius, 10.0
%area, 314.159265

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

**注意：** `*PARAMETER` 定义行中的 `%area, %pi * %radius^2` 被替换为 `%area, 314.159265`。但 `%pi` 和 `%radius` 的定义行保持不变（因为它们不是表达式，已经是纯数值）。

**另一个例子 — 内建函数：**

```
*PARAMETER
%angle, 30
%rad, %angle * 3.14159265 / 180

*LOAD_BODY_Z
1, sin(%rad)
```

执行 `:re -b` 后：

```
*PARAMETER
%angle, 30
%rad, 0.523598775

*LOAD_BODY_Z
1, 0.5
```

**支持的内建函数列表：**

| 函数 | 说明 |
|------|------|
| `sin` / `cos` / `tan` | 三角函数（角度制） |
| `sinr` / `cosr` / `tanr` | 三角函数（弧度制） |
| `asin` / `acos` / `atan` | 反三角函数（角度制） |
| `asinr` / `acosr` / `atanr` | 反三角函数（弧度制） |
| `tanh` | 双曲正切 |
| `exp` / `ln` / `log` / `log10` | 指数和对数 |
| `sqrt` | 平方根 |
| `abs` | 绝对值 |
| `sign` | 符号函数 |
| `floor` / `ceil` / `round` | 取整 |
| `mod` | 取模 |
| `min` / `max` | 最小/最大值 |
| `H` | Heaviside 阶跃函数 |
| `d` | Dirac delta 函数近似 |
| `erf` | 误差函数 |
| `pi` | 圆周率常量 |

**适用场景：** 你的输入文件包含复杂数学表达式，需要完全展开为数值后提交给求解器。

**`:Re -b` 和 `:re -b` 的区别：**
- 两者功能完全相同
- 输入 `:re -b` 时，Neovim 的 `cnoreabbrev` 会自动把 `re` 展开为 `Re`
- 所以实际执行的是 `:Re -b`

---

### 10.4 `:re -c` — 带循环溢出检测

**功能：** 同 `:re -b`，但增加了对 `~repeat` 块循环溢出的检测。如果 `~repeat` 的展开次数超过安全阈值，会发出警告。

**适用场景：** 你的文件包含大量 `~repeat` 展开，想避免意外生成过大的文件。

---

### 10.5 Replace 命令的安全限制与防护机制

`:re`、`:re -a`、`:re -b` 在替换和求值过程中设置了多层安全防护，防止因循环引用、参数链膨胀或大文件导致的内存耗尽或 Neovim 崩溃。

#### 循环引用检测（防止死循环）

如果参数之间相互引用形成闭环，例如：

```
*PARAMETER
%a = %b
%b = %a
```

插件会检测到这个循环并立即中止替换，弹出警告：

```
Circular parameter reference detected: %a, %b. Replace aborted.
```

**实现原理**：`substitute_vars` 内部维护一个 `chain` 表，记录当前正在展开的参数名。如果在递归展开过程中再次遇到链表中已存在的参数名，即判定为循环引用。最大递归深度限制为 **15 层**。

#### 字符串长度保险丝（防止溢出）

即使不存在循环，一个长参数引用链也可能产生极长的展开字符串：

```
*PARAMETER
%a1 = 1
%a2 = %a1 + 1
%a3 = %a2 + 1
…
%a1000 = %a999 + 1
```

`%a1000` 的展开结果会变成 `"1 + 1 + … + 1"`（约 2000 字符）。如果一行内同时引用多个这样的长参数，字符串长度会急剧膨胀。

插件在三处设置了 **5000 字符** 的长度上限：

| 位置 | 作用 |
|------|------|
| `substitute_vars` | 参数替换的中间结果超过 5000 字符时中止 |
| `substitute_in_context` | `build_param_tables` 阶段冻结参数值时超过 5000 字符时保留原值 |
| `simplify_numeric_text` | 单行长度超过 5000 字符时跳过算术简化 |

当任何一处保险丝触发时，替换操作会安全中止，不会导致 `not enough memory` 错误。

#### 大文件内存保护（定期垃圾回收）

`simplify_numeric_text_fixed_point` 对每行最多迭代 4 次，每次迭代对每个字段和方括号表达式调用求值器。`:re -b` 使用的 `eval_expr_with_functions` 每次调用会创建约 8 个 Lua 闭包。在 10 000 行以上的大文件中，闭包数量可能达到数十万，导致 Lua 堆内存碎片化。

插件在替换过程中**每处理 1000 行强制触发一次完整垃圾回收**（`collectgarbage("collect")`），及时释放闭包和临时字符串，防止内存分配失败。

#### 参数定义行不在第二遍中重复处理

`:re` 命令采用**两遍处理**：
1. **第一遍**：逐行扫描，遇到 `*PARAMETER` 定义行时求值参数右侧表达式并更新 `current_vars`；对非参数行替换 `%name` 和 `[expr]`。
2. **第二遍**：对所有非跳过行调用 `simplify_numeric_text_fixed_point`，进一步简化嵌套数值表达式。

参数定义行（如 `R11 = 0`）已在第一遍中处理完毕，第二遍再次将其传给算术简化器会产生误报：`R11` 被识别为未知标识符。因此**第二遍会跳过所有参数定义行**。

---

### 10.6 Replace 命令的特殊行为

**跨文件参数替换：**

`:re`、`:re -a`、`:re -b` 都会自动递归解析当前文件中的所有 `*INCLUDE` 引用，收集子文件中定义的参数用于替换。

**举例：**

`materials.k`:
```
*PARAMETER
%a, 1
%b, %a / 2
```

`main.k`:
```
*INCLUDE
materials.k

*MAT_ELASTIC
1, %b, 70.0e9, 0.3
```

在 `main.k` 中执行 `:re -a`：
- `%b` 被替换为 `0.5`（因为 `materials.k` 中 `%a = 1`，所以 `%b = 1 / 2 = 0.5`）
- 即使 `main.k` 后面覆盖了 `%a = 23`，`%b` 仍然保持 `0.5`

**参数重新赋值（覆盖）：**

同一个文件中，后面的参数定义会覆盖前面的。替换引擎严格按照**定义顺序**逐步求值。

```
*PARAMETER
%a, 1
%b, %a / 2       ← %b = 0.5（使用 %a = 1）
%a, 23
%c, %a * 2       ← %c = 46（使用 %a = 23）
```

执行 `:re -a` 后：
```
*PARAMETER
%a, 1
%b, 0.5
%a, 23
%c, 46
```

**注意：** 如果 `:re` 看起来没有替换某些参数，检查：
1. 参数格式是否正确（支持 `%name, value` 和 `%name = value`）
2. `*INCLUDE` 路径是否正确（不可读的文件会显示警告）
3. 参数是否定义在 `~if` 条件块内（当前不支持条件解析）

---

**`*FUNCTION` 块的处理：**

Replace 命令会**跳过** `*FUNCTION` 块内的数据行（函数表达式），不会替换其中的参数。这是因为函数表达式是求解器在运行时求值的，不应该被静态替换。

**举例：**

```
*FUNCTION
"Velocity profile"
101
10.0 * sin(2 * 3.14159265 * t)

*MAT_ELASTIC
1, %density, 70.0e9, 0.3
```

执行 `:re -b` 后：

```
*FUNCTION
"Velocity profile"
101
10.0 * sin(2 * 3.14159265 * t)

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

`*FUNCTION` 的表达式行保持不变，`*MAT_ELASTIC` 的 `%density` 被替换。

**`*FUNCTION` 隐式结束检测：**

如果 `*FUNCTION` 块缺少 `*END_FUNCTION`，插件会自动检测下一个顶层关键字作为块的结束。这修复了之前 `:re -b` 会把后续数据行误判为函数表达式而跳过的 bug。

---

## 11. Lint 检查命令详解

Lint 命令对当前文件进行全面的模型验证，发现潜在错误。主命令是 `:Cc`（或 `:Ccheck`、`:Chk`）。执行完成后，光标会自动跳转到最严重的问题处（Error → Warning → Suspicion）。

### 11.1 运行方式

| 命令 | 说明 |
|------|------|
| `:Cc` | 运行所有检查 |
| `:Ccheck` | 同 `:Cc` |
| `:Chk` | 同 `:Cc` |

保存文件时如果 `lint_on_save = true`（默认），会自动运行 `:Cc`。

执行 `:clean`（无参数）可以清除所有 Lint 诊断标记。

### 11.2 三级严重程度

Lint 结果分为三个级别：

| 级别 | 颜色 | 含义 | 是否必须修复 |
|------|------|------|-------------|
| **Error** | 🔴 红色 | 明确的错误，求解器会报错或崩溃 | **必须修复** |
| **Warning** | 🟡 黄色 | 可疑内容，可能导致非预期行为 | **建议修复** |
| **Suspicion** | 🔵 蓝色（带 `?`） | 超出物理合理范围，需要人工确认 | 审查后决定 |

### 11.3 13 项检查详解

#### 检查 1: 控制块平衡（Control Block Balance）— Error

**检查内容：** 所有 `~if` / `~end_if`、`~repeat` / `~end_repeat`、`~convert_from_*` / `~end_convert`、`~begin_scope` / `~end_scope` 是否成对出现。

**举例：**

```
~if %use_gravity == 1
*LOAD_BODY_Z
1, 9.81
```

缺少 `~end_if`。`:Cc` 报告：

```
E1001: Unmatched ~if at line 42
```

**修复：** 添加 `~end_if`。

---

#### 检查 2: 未知关键字（Unknown Keywords）— Warning

**检查内容：** 文件中出现了不在 `keywords.json` 数据库中的关键字。

**举例：**

```
*MY_CUSTOM_KEYWORD
1, 2, 3
```

`:Cc` 报告：

```
W1002: Unknown keyword "*MY_CUSTOM_KEYWORD" at line 15
```

**注意：** 这只是一个警告，因为你的自定义关键字可能在求解器中是合法的，只是插件不认识。

---

#### 检查 3: 字段数量不匹配（Field Count Mismatch）— Error

**检查内容：** 关键字的第一行数据字段数量是否与 `commands.help` 中定义的签名不匹配。

**举例：** `*MAT_ELASTIC` 的定义是 `mid, rho, E, nu`（4 个字段）。

```
*MAT_ELASTIC
1, 2700.0, 70.0e9
```

只有 3 个字段，缺少 `nu`。`:Cc` 报告：

```
E1003: Field count mismatch for *MAT_ELASTIC at line 23. Expected 4, got 3
```

**修复：** 补充缺失的字段，或用 `-` 占位。

---

#### 检查 4: 未定义参数引用（Undefined Parameter References）— Error

**检查内容：** `%param` 在当前文件及其所有 `*INCLUDE` 引用的文件中找不到定义。

**举例：**

```
*MAT_ELASTIC
1, %density, 70.0e9, 0.3
```

但没有 `*PARAMETER` 定义 `%density`。`:Cc` 报告：

```
E1004: Undefined parameter "%density" at line 45
```

**修复：** 在 `*PARAMETER` 中添加定义，或检查拼写错误。

---

#### 检查 5: 未使用参数（Unused Parameters）— Warning

**检查内容：** `*PARAMETER` 中定义了参数，但文件中没有任何地方引用它。

**举例：**

```
*PARAMETER
%density, 2700.0
%unused, 100.0

*MAT_ELASTIC
1, %density, 70.0e9, 0.3
```

`%unused` 定义了但从未使用。`:Cc` 报告：

```
W1005: Unused parameter "%unused" at line 3
```

**修复：** 删除未使用的参数定义，或检查是否遗漏了引用。

---

#### 检查 6: 重复对象 ID（Duplicate Object IDs）— Error

**检查内容：** 同一类型的对象 ID 在同一个文件中重复定义。

**举例：**

```
*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3

*MAT_ELASTIC
1, 7850.0, 210.0e9, 0.3
```

`mid=1` 定义了两次。`:Cc` 报告：

```
E1006: Duplicate MAT ID 1 at lines 10 and 15
```

**注意：** 跨文件重复是允许的（LS-DYNA 允许不同 `*INCLUDE` 文件中使用相同 ID）。

---

#### 检查 7: 缺失的 `*INCLUDE` 文件（Missing Include Files）— Error

**检查内容：** `*INCLUDE` 引用的文件路径不存在。

**举例：**

```
*INCLUDE
../mesh/nonexistent.k
```

`:Cc` 报告：

```
E1007: Include file not found: "../mesh/nonexistent.k" at line 8
```

---

#### 检查 8: 空关键字块（Empty Keyword Blocks）— Warning

**检查内容：** 关键字块只有关键字行，没有任何数据行。

**举例：**

```
*MAT_ELASTIC

*NODE
```

`:Cc` 报告：

```
W1008: Empty keyword block "*MAT_ELASTIC" at line 20
```

---

#### 检查 9: 未定义对象引用（Undefined Object References）— Error / Warning

**检查内容：** 引用了不存在或未定义的对象 ID。

**举例：**

```
*PART
"Front bumper"
1, 1, 999, 0, 0
```

`mid=999` 未定义。`:Cc` 报告：

```
E1009: Undefined MAT reference 999 in *PART at line 30
```

**零值特殊处理：** `0` 被视为"未设置/未使用"，不会报错。例如：

```
*PART
"Front bumper"
1, 1, 1, 0, 0
```

最后两个 `0` 是 `eosid=0` 和 `hgid=0`，不会被标记为错误。

---

#### 检查 10: 必填字段为空（Required Fields Empty）— Error

**检查内容：** 关键字签名中标记为必填的字段为空或只有 `-`。

**举例：**

```
*MAT_ELASTIC
1, -, 70.0e9, 0.3
```

`rho` 字段为 `-`。`:Cc` 报告：

```
E1010: Required field "rho" is empty in *MAT_ELASTIC at line 25
```

---

#### 检查 11: 枚举值无效（Invalid Enum Values）— Error

**检查内容：** 字段的值不在 `commands.help` 定义的合法选项中。

**举例：**

```
*UNIT_SYSTEM
INVALID, K
```

`:Cc` 报告：

```
E1011: Invalid enum value "INVALID" for field "units" in *UNIT_SYSTEM at line 5
```

**合法值示例：** `*UNIT_SYSTEM units` 支持 `SI`、`MMTONS`、`CMGUS`、`IPS` 等 14 种。

---

#### 检查 12: 物理合理性（Physics Sanity）— Suspicion

**检查内容：** 数值是否落在物理合理范围内。根据 `*UNIT_SYSTEM` 自动选择对应的合理范围。

**举例：**

```
*UNIT_SYSTEM
SI, K

*MAT_ELASTIC
1, 2700.0, 70.0e9, 0.3
```

`rho = 2700.0 kg/m³` 对于 SI 单位是合理的（铝的密度）。但如果：

```
*MAT_ELASTIC
1, 2.7e12, 70.0e9, 0.3
```

`rho = 2.7e12 kg/m³` 远超任何已知材料。`:Cc` 报告：

```
?1012: Density 2.7e12 is suspiciously high for SI units at line 15
```

**检查的项目包括：**
- 密度（Density）
- 杨氏模量（Young's Modulus）
- 长度（Length）
- 速度（Velocity）
- 质量（Mass）

**14 个支持的单位系统：** `SI`、`MMTONS`、`CMGUS`、`IPS`、`MMKGMS`、`CMGS`、`MMGMS`、`MMMGMS` 及其别名。

---

#### 检查 13: 缺少单位系统（Missing Unit System）— Warning

**检查内容：** 文件中没有 `*UNIT_SYSTEM` 定义。

**举例：** 文件中没有任何 `*UNIT_SYSTEM` 行。`:Cc` 报告：

```
W1013: No unit system defined in file
```

**影响：** 没有单位系统时，物理合理性检查（检查 12）无法运行。

---

#### 检查 14: `*CURVE` / `*FUNCTION` X 坐标递增 — Error

**检查内容：** `*CURVE` 和 `*FUNCTION` 数据点的 x 坐标必须严格递增。

**举例：**

```
*CURVE
"Force-displacement"
101
0.0, 0.0
0.1, 100.0
0.05, 200.0
0.3, 300.0
```

第 3 行 `0.05` 小于第 2 行 `0.1`。`:Cc` 报告：

```
E1014: X-coordinate not ascending at line 6: 0.05 <= 0.1
```

**智能跳过：** 对于 `*CURVE`，插件会自动检测数据起始行，跳过 title 行和 ID 行。

**支持的分隔符：** 逗号分隔 `0.0, 0.0` 和空格分隔 `0.0 0.0` 都可以识别。

---

### 11.4 Lint 诊断的显示方式

Lint 结果通过 Neovim 的 `vim.diagnostic` API 显示：

- **虚拟文本（Virtual Text）**：在行尾显示错误代码和描述
- **标记列（Sign Column）**：左侧显示 `E`、`W`、`?` 标记
- **下划线（Underline）**：错误处带红色/黄色/蓝色波浪下划线

**在 quickfix 中查看所有诊断：**

```vim
:Cc
:copen
```

> 提示：`:Cc` 执行后会自动跳转到最高优先级的诊断位置。如果你不希望跳转，可以在运行 `:Cc` 前设置标记（如 `ma`），之后用 `` `a `` 回到原来位置。

### 11.5 自动运行

如果 `lint_on_save = true`（默认），保存文件（`:w`）时会自动运行 `:Cc`。

如果文件很大或包含很多 `*INCLUDE`，Lint 可能需要几百毫秒。你可以临时禁用：

```lua
require("impetus").setup({ lint_on_save = false })
```

## 15. 补全系统详解

插件提供多层级的上下文感知补全，适用于关键字、参数、对象 ID 和枚举选项。

### 15.1 关键字补全

**触发方式：**
- 在关键字行开头输入 `*`（如 `*MAT`）
- 按 `<C-Space>` 手动触发

**行为：**
- 弹出 Blink.cmp 菜单，显示匹配的关键字列表
- 选择后展开为 snippet 模板，包含所有数据字段的占位符

## 16. 参数系统详解

### 16.1 参数定义

参数通过 `*PARAMETER` 或 `*PARAMETER_DEFAULT` 定义。插件支持**两种格式**：

**格式 A — 逗号分隔（Impetus 标准格式）：**

```
*PARAMETER
%density, 2700.0
%young, 70.0e9
%poisson, 0.3
%size, 10 + 5
```

**格式 B — 等号分隔：**

```
*PARAMETER
%density = 2700.0
%young = 70.0e9
%poisson = 0.3
%size = 10 + 5
```

两种格式完全等价，插件会自动识别。值可以是：
- 纯数字：`2700.0`
- 表达式：`10 + 5`
- 引用其他参数：`%size * 2`（需要 `:re -b` 才能完全展开）

---

### 16.1a 参数重新赋值（覆盖）— 定义顺序至关重要 ⚠️

**参数可以在同一个文件中多次定义，后面的定义会覆盖前面的。** 插件严格按照**从上到下的定义顺序**逐步求值，每个参数的值在定义那一刻就被冻结。

**举例：**

```
*PARAMETER
%a, 1
%b, %a / 2
%a, 23
%c, %a * 2
```

**求值结果：**

| 参数 | 求值时 `%a` 的值 | 最终值 |
|------|------------------|--------|
| `%a`（第一次） | — | `1` |
| `%b` | `%a = 1` | `1 / 2 = 0.5` |
| `%a`（第二次，覆盖） | — | `23` |
| `%c` | `%a = 23` | `23 * 2 = 46` |

**执行 `:re -a` 后：**

```
*PARAMETER
%a, 1
%b, 0.5
%a, 23
%c, 46
```

**关键规则：**
- `%b` 使用的是**第一次** `%a = 1` 的值，而不是最后被覆盖的 `23`
- 如果 `%b` 的定义写在 `%a = 23` 之后，那么 `%b` 才会使用 `23`
- **定义顺序决定求值结果**，不是字母顺序，也不是最后出现的值

**错误示例（不要这样做）：**

```
*PARAMETER
%a, 23        ← 先定义 23
%b, %a / 2    ← %b = 11.5
%a, 1         ← 覆盖为 1，但 %b 已经是 11.5 了！
```

如果你期望 `%b = 0.5`，必须确保 `%a = 1` 在 `%b` 的定义之前。

---

### 16.2 参数引用

在数据行中使用 `%名称` 引用参数：

```
*MAT_ELASTIC
1, %density, %young, %poisson
```

---

### 16.3 方括号表达式

支持 `[%表达式]` 形式的 inline 计算：

```
*NODE
1, [%L / 2], 0.0, 0.0
```

其中 `%L` 是一个参数。`:re -a` 或 `:re -b` 会把 `[%L / 2]` 替换为计算后的数值。

---

### 16.4 参数作用域

- **文件内参数：** 在 `*PARAMETER` 块中定义，只在本文件有效
- **跨文件参数：** 如果文件 A 通过 `*INCLUDE` 引用了文件 B，文件 B 中定义的参数在文件 A 中也可见

**跨文件参数在 `*INCLUDE` 文件中的冻结规则：**

当 `*INCLUDE` 文件定义了依赖参数（如 `%b = %a / 2`），`%b` 的值在 `*INCLUDE` 文件内部就被冻结了。即使主文件后面覆盖了 `%a`，也不会影响 `%b` 的值。

**举例：**

`materials.k`:
```
*PARAMETER
%a, 1
%b, %a / 2     ← 在 materials.k 内部求值，%b = 0.5
```

`main.k`:
```
*INCLUDE
materials.k

*PARAMETER
%a, 23
%c, %a * 2     ← %c = 46（使用 main.k 中的 %a = 23）

*PART
"Part A"
1, 1, 1, 0, 0
```

执行 `:re -a` 后：
- `main.k` 中的 `%a` 被替换为 `23`
- `main.k` 中的 `%c` 被替换为 `46`
- 但 `%b` 保持为 `0.5`（因为它在 `materials.k` 中已经被冻结为 `1 / 2`）

**导航：** 在 `main.k` 中，`:gd` 放在 `%density` 上，会跳转到 `materials.k` 中的定义。

**如果跨文件参数没有替换：** 检查 `*INCLUDE` 路径是否正确，以及 `materials.k` 中的参数格式是否被支持（逗号或等号格式）。如果文件不可读，Neovim 会显示警告信息。

---

### 16.5 特殊参数语义

**零值 = 未设置：**

在 Impetus 中，`0` 通常表示"不使用/未设置"，而不是字面量的零。插件的 Lint 检查对此有特殊处理：

```
*PART
"Front bumper"
1, 1, 1, 0, 0
```

最后两个 `0` 是 `eosid` 和 `hgid`。Lint **不会**把它们报告为"未定义的对象引用"，因为 `0` 是合法的"未使用"标记。

## 19. 跨文件参数解析

插件通过递归扫描 `*INCLUDE` 文件，建立跨文件的参数和对象索引。

### 19.1 解析机制

1. 解析当前文件，找到所有 `*INCLUDE` 行
2. 递归打开引用的文件，解析其中的 `*PARAMETER`、`*PART`、`*MAT_` 等定义
3. 建立全局索引：参数名 → 定义位置，对象 ID → 定义位置

### 19.2 使用场景

**场景 1 — 参数定义在子文件中：**

`main.k`:
```
*INCLUDE
materials.k

*PART
"Part A"
1, 1, 1, 0, 0
```

`materials.k`:
```
*PARAMETER
%density, 2700.0
*MAT_ELASTIC
1, %density, 70.0e9, 0.3
```

在 `main.k` 中：
- `:gd` 放在 `%density` 上 → 跳转到 `materials.k`
- `:gr` 放在 `MAT 1` 上 → 显示 `main.k` 中的 `PART 1` 引用

**场景 2 — 对象引用跨文件：**

`main.k` 中的 `*PART` 引用了 `mid=1`，而 `*MAT_ELASTIC` 定义在 `materials.k` 中。Lint 检查时，插件会正确识别这是合法的引用，不会报告"未定义对象"。

### 19.3 性能考虑

- 大型 `*INCLUDE` 树（几十个子文件）的首次解析可能需要 100-300ms
- 解析结果会缓存，后续操作（如 `gd`、`gr`、`:Cc`）会复用缓存
- 保存文件后缓存自动失效并重新构建

## 20. 配置选项

完整的 `setup()` 配置选项：

```lua
require("impetus").setup({
  -- 数据库文件
  help_file = "/path/to/commands.help",     -- 关键字数据库源文件
  cache_file = nil,                          -- 缓存数据库路径（自动检测）
  auto_load = true,                          -- 启动时自动加载

  -- Lint 设置
  lint_on_save = true,                       -- 保存时自动运行 :Cc

  -- 快捷键设置
  tab_field_jump = true,                     -- <Tab> 在参数字段间跳转
  blink_menu_keys = true,                    -- Blink.cmp 菜单用 j/k/Space

  -- 引用标记
  ref_marks_enabled = true,                  -- 显示对象引用标记（绿/蓝下划线）

  -- 开发模式
  dev_hot_reload = false,                    -- 焦点返回时热重载
  dev_mode = false,                          -- 启用调试命令
})
```

### 20.1 配置项详解

| 选项 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `help_file` | string | `nil` | `commands.help` 文件的完整路径。如果提供，插件会优先从此文件加载数据库 |
| `cache_file` | string | `nil` | JSON 缓存文件路径。如果 `help_file` 已加载且缓存有效，优先使用缓存加速启动 |
| `auto_load` | boolean | `true` | 启动时是否自动加载数据库 |
| `lint_on_save` | boolean | `true` | 保存文件时是否自动运行 Lint |
| `tab_field_jump` | boolean | `true` | 插入模式下 `<Tab>` 是否在 snippet 占位符间跳转 |
| `blink_menu_keys` | boolean | `true` | Blink.cmp 补全菜单是否使用 `j`/`k`/`Space` 导航 |
| `ref_marks_enabled` | boolean | `true` | 是否默认显示引用标记（可后续用 `:Crm` 切换） |
| `dev_hot_reload` | boolean | `false` | 窗口获得焦点时是否自动重载插件 Lua 文件（开发调试用） |
| `dev_mode` | boolean | `false` | 启用开发调试命令（`:Cdbg`、`:Cdoctor` 等） |

## 21. 推荐工作流

### 21.1 日常编辑工作流

```
1. 打开 .k 文件
2. 按 ,h 打开帮助面板（可选）
3. 编辑数据
   - 用 * + Tab 插入新关键字
   - 用 ,, 选择枚举值或对象 ID
   - 用 <Tab> 在字段间跳转
4. 保存 :w（自动运行 Lint）
5. 修复 Lint 错误（红色标记）
6. 按 ,c 注释/取消注释块进行调试
```

### 21.2 参数化建模工作流

```
1. 在文件顶部定义参数：
   *PARAMETER
   %density, 2700.0
   %young, 70.0e9

2. 在数据行中使用 %参数

3. 保存并检查 :Cc

4. 需要提交给求解器时：
   :re -b    → 展开所有参数和表达式
   :clean -a → 清理并格式化
```

### 21.3 模型审查工作流

```
1. :Ccheck 或保存文件 → 运行 Lint
2. 修复所有 Error（红色）
3. 审查所有 Warning（黄色）
4. 审查 Suspicion（蓝色带 ?）
   - 确认物理数值是否合理
   - 单位系统是否匹配
5. :Cgraph → 查看对象统计
6. :Cgdel MAT:1 → 检查是否可以删除未使用的材料
```

### 21.4 跨文件项目工作流

```
1. 主文件用 *INCLUDE 引用子文件
2. 参数定义在公共文件（如 parameters.k）
3. 在主文件中：
   - :gd 跳转跨文件定义
   - :gr 查找跨文件引用
   - :Cc 自动检查所有引用文件
4. :Cgeo → 预览整体几何
```

## 22. 故障排除

### 22.1 常见问题

**Q: 插件没有加载，按 `,c` 没反应**

A: 检查以下几点：
1. 文件扩展名是否为 `.k` 或 `.key`
2. `:set filetype?` 是否显示 `impetus`
3. 检查 `help_file` 配置路径是否正确
4. 运行 `:ImpetusReload` 尝试重新加载

---

**Q: `:Cc` 报很多 "Unknown keyword" 警告**

A: 这些是你的自定义关键字，插件不认识。如果求解器支持这些关键字，可以忽略这些警告。如果想消除警告，需要在 `commands.help` 中添加这些关键字的定义。

---

**Q: `gd` 跳转到错误位置**

A: 可能的原因：
1. 对象 ID 重复定义 — 插件跳转到第一个定义
2. 跨文件引用未解析 — 检查 `*INCLUDE` 路径是否正确
3. 参数定义在条件块（`~if`）内 — 当前不支持条件解析

---

**Q: `:re -b` 后 `*FUNCTION` 块的内容被修改了**

A: 正常情况下 `*FUNCTION` 块应该被跳过。如果发生这种情况：
1. 确认 `*FUNCTION` 块有明确的结束标记（`*END_FUNCTION` 或下一个顶层关键字）
2. 更新插件到最新版本（已修复隐式结束检测）

---

**Q: 几何预览器无法启动**

A: 检查以下几点：
1. Python 是否安装且 `python.exe` 在 PATH 中
2. `scripts/impetus_geometry_viewer.py` 是否存在
3. 是否有 Tkinter 和 VTK 库：`pip install vtk`

---

**Q: 补全菜单不弹出**

A: 检查以下几点：
1. 是否在插入模式
2. 光标是否在关键字行开头（`^%s*%*` 匹配）
3. Blink.cmp 是否正确配置
4. 数据库是否已加载（`:ImpetusReload`）

---

**Q: 折叠状态丢失了**

A: 折叠状态只在当前 Neovim 会话内保持。关闭 Neovim 后状态会丢失。可以通过 `:mksession` 保存会话恢复窗口布局。

---

**Q: `:clean -a` 删除了我需要的内容**

A: `:clean -a` 会删除未知关键字块。**不可逆！** 建议：
1. 先运行 `:clean -c` 查看哪些块会被标记为"未知"
2. 如果确认不需要未知块，再运行 `:clean -a`
3. 或者使用版本控制（Git），出错后可以恢复

---
